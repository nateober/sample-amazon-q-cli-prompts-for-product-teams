Overview

The Product Team's Tool Chest transforms AI coding assistants into comprehensive product development partners. Using steering files, the toolkit guides AI through a complete workflow from market research to interactive prototype.

What is Kiro?

Kiro is an AI-powered IDE that uses "steering files" to guide AI behavior with project-specific instructions. Steering files are markdown documents in .kiro/steering/ that provide context, workflows, and constraints. When you work in Kiro, these files automatically shape how the AI assistant responds—no manual prompting required.

The 4-Phase Workflow

1

Market Research

AI conducts web-based competitive analysis, market sizing, and customer pain point research.

2

PRFAQ

Amazon-style "Working Backwards" documentation with Press Release and FAQ sections.

3

PRD

Implementation-ready requirements in EARS syntax with personas and user stories.

4

Prototype

Interactive HTML prototypes with modular screens and distinctive design.

All Outputs are Standalone HTML

Every document is a self-contained HTML file that opens directly in any browser. No build step, no server, no dependencies beyond Google Fonts. Share with stakeholders by simply sending the files.

Setup Guide

Option A: Kiro (Recommended)

The fastest way to get started. Kiro natively supports steering files.

Step 1: Install Kiro

Download from kiro.dev

Step 2: Copy Steering Files

# From the toolkit directory cp -r .kiro/steering/* your-project/.kiro/steering/ cp .kiro/hooks.json your-project/.kiro/

Step 3: Open in Kiro and Describe Your Product

That's it! The steering files automatically guide the AI through the 4-phase workflow.

Option B: Legacy Mode

If your tool doesn't support steering files, use the prompts folder directly.

Legacy Mode Limitations

  • No automated market research (web search may not be available)
  • Manual prompt loading for each phase
  • No automatic validation hooks

Legacy Setup Steps

  1. Copy the prompts/ folder to your project
  2. Start your AI tool in the project directory
  3. Load guides manually for each phase

The 4-Phase Workflow

Each phase produces validated HTML documents. The workflow includes automatic validation checkpoints—it won't proceed until each phase passes quality checks.

Phase 1: Market Research

AI automatically conducts:

  • Competitive landscape analysis (3-5 competitors)
  • Market sizing with TAM/SAM/SOM and cited sources
  • Customer pain points from reviews and forums
  • Pricing intelligence and benchmarks
  • Customer brand research (logo, colors) if building for a company

Output: MarketResearch_[Product]_[Date].html

Validation Requirements

  • TAM/SAM/SOM with actual dollar figures and sources
  • At least 3 competitors with real pricing data
  • Pain points must be specific, not generic
  • No placeholder text (TBD, TODO, [insert])

Phase 1b: AI Framing (Optional)

Only for AI/ML products. Defines ML problem type, input/output schemas, success metrics, and training data requirements.

Phase 2: PRFAQ

Amazon-style Working Backwards documentation:

  • Press Release written as if product already launched
  • FAQ addressing customer and business questions
  • Incorporates market research findings

Output: PRFAQ_[Product]_[Date].html

Phase 3: PRD

Implementation-ready specification:

  • User personas with detailed profiles
  • Requirements in EARS syntax (When/The/Shall format)
  • User stories with acceptance criteria
  • Success metrics and business model

Output: PRD_[Product]_[Date].html + Kiro spec files

Tech Stack Constraints

Allowed: Lambda, ECS, EC2, App Runner, DynamoDB, Aurora, RDS, Amazon Bedrock, Claude API, S3, EFS, API Gateway, AppSync, Cognito

Not Allowed: OpenAI, Google Cloud, Azure, Vercel, Firebase, Supabase

Phase 4: Prototype

Interactive HTML prototype with:

  • Modular file structure (NOT a single monolithic file)
  • Distinctive visual design (no generic "AI slop")
  • Working navigation between screens
  • Form validation and feedback
  • Responsive layouts (desktop, tablet, mobile)
  • Realistic data (no Lorem ipsum)
documents/ ├── DesignSystem_[Product]_[Date].html (shared CSS - created FIRST) ├── ScreenIndex_[Product]_[Date].html (navigation hub) ├── Screen_Dashboard_[Product]_[Date].html ├── Screen_[Name]_[Product]_[Date].html (one per screen) └── ProjectDashboard_[Product]_[Date].html

Design Standards

The toolkit enforces distinctive design to avoid generic "AI slop" aesthetics.

Purple-to-Blue Gradients

The most overused AI aesthetic

Dominant, secondary, and accent colors

Gradients, shadows, and depth

Using framework defaults looks lazy

Best Practices

Trust the Validation

The workflow won't proceed until validation passes. If you're tempted to skip validation, that's a sign the work needs improvement.

Provide Rich Context

The more real data you provide (team members, customer info, brand guidelines), the better the output quality.

Don't Skip Market Research

The PRFAQ and PRD are dramatically better when grounded in real competitive data. Market research is mandatory in the Kiro workflow.

Modular Prototypes Only

Never accept a single monolithic HTML file. The prototype must be split into separate screen files with a shared design system.
